# Bug Report: 500 Error on Workout Update via `gc` CLI

## Description
Attempting to update an existing workout using the `gc workouts update` command fails with a `500 Internal Server Error` from the Garmin Connect API. This suggests an issue with the API endpoint, the payload structure for updates, or a temporary service outage, although the error "too many 500 error responses" implies a persistent failure for this request.

## Environment
- **Tool:** `garmin-cli` (via `gc` command)
- **API Endpoint:** `https://connectapi.garmin.com/workout-service/workout/{workout_id}`
- **Method:** `PUT`

## Reproduction Steps
Run the following command to update a workout with specific steps and targets:

```bash
gc workouts update 1477820256 \
  --name "Lumi: 2h Endurance + SST (Targeted)" \
  --sport cycling \
  --steps '[
    {"type":"warmup","duration":900},
    {"type":"interval","duration":1200,"target":"power_zone:3"},
    {"type":"recovery","duration":600,"target":"power_zone:1"},
    {"type":"interval","duration":1200,"target":"power_zone:3"},
    {"type":"cooldown","duration":3300,"target":"power_zone:2"}
  ]' \
  --format json
```

## Error Log
```text
urllib3.exceptions.ResponseError: too many 500 error responses

The above exception was the direct cause of the following exception:

Traceback (most recent call last):
  ...
  File "garmincli/commands/workouts.py", line 561, in update
    data = _workout_request(
        client, "PUT", f"/workout-service/workout/{workout_id}", payload
    )
  ...
requests.exceptions.RetryError: HTTPSConnectionPool(host='connectapi.garmin.com', port=443): Max retries exceeded with url: /workout-service/workout/1477820256 (Caused by ResponseError('too many 500 error responses'))
```

## Analysis
- The payload uses shorthand step definitions (`type`, `duration`, `target`).
- The `target` field uses the format `power_zone:X`.
- It is possible that the `update` endpoint requires a different payload structure compared to `create`, or that specific fields (like `stepOrder` or full object structures) are strictly required for updates but inferred for creation.
- Alternatively, the specific workout ID might be in a state that prevents updates.

## Suggested Investigation
1.  Verify if `gc workouts create` works with the exact same payload (creation vs. update).
2.  Inspect the full JSON payload generated by the CLI before sending.
3.  Check if `stepOrder` needs to be explicit in updates.
